\section{Failure Case: en-fr on the Loose-Filter v2 Corpus}
\label{sec:enfr_v2}

This section reports the failure side of the 2$\times$2 ablation. Both Base v2 and Big v2 are trained on the same pipeline, same SPM, same eval, same hyperparameters as their v1.1 counterparts; only the training corpus changes (v1's 9.3M strict-filter $\rightarrow$ v2's 30M loose-filter, drawn from the same WMT14 source pool). Both v2 runs converge cleanly. \emph{Both land below their v1 counterparts}, and \emph{Big v2 lands below Base v2} on the same data. The 2$\times$2 closes here.

\subsection{Corpus construction}
v2 is the same WMT14 en-fr raw $\sim$~40.8M, cleaned with the same per-pair filters as v1, but \emph{without} the 10M-head cap. 30.1M pairs survive (74\% keep rate, vs.\ v1's 93\% rate on the head 10M). The 19\%-point keep-rate difference quantifies how much noisier the long-tail data is once the cap is removed.

A schedule re-tuning is required because v2 is $3.2\times$ larger: we set Base v2 to 600K steps (vs.\ Base v1.1's 122K) and Big v2 to 800K (vs.\ Big v1.1's 280K), maintaining the same $\sim$~3 epochs. Warmup is scaled proportionally (4K~$\rightarrow$~12K for Base, 8K~$\rightarrow$~24K for Big). All other hyperparameters are unchanged.

\subsection{Base v2: 60M parameters, 30M loose data}
\label{sec:base_v2}
Trained for 600K steps (4 epochs over 30M); converged best valid 28.66, averaged 5 checkpoints valid 29.23 / test 33.90. Total wall-clock: $\sim$~14~h.

This is \textbf{below Base v1.1} on both splits: $-1.29$ valid, $-1.41$ test. Despite seeing $3.2\times$ more data and running $4.9\times$ more optimizer steps, v2 underperforms v1.1 by more than 1 full BLEU.

\subsection{Big v2: 209M parameters, 30M loose data}
\label{sec:big_v2}
Trained from scratch for 800K planned steps; halted at 416K when valid BLEU plateaued and a single eval dipped below the previous best. Best single-checkpoint valid 27.77; averaging 5 checkpoints in [345K, 416K] gives valid 28.91 / test 33.03. Total wall-clock: $\sim$~9~h to halt.

This is \textbf{below Base v2} on both splits: $-0.32$ valid, $-0.87$ test. The same $3.5\times$ extra parameters that paid $+0.56$ on v1 (\S\ref{sec:v1_capacity}) cost $-0.87$ here. The capacity gap has flipped sign.

Table~\ref{tab:two_by_two} is the central empirical claim of this paper. Two ways to read it:
\begin{enumerate}
\item \textbf{Row-wise (capacity gap):} on clean data Big beats Base by $+0.56$; on noisy data Big falls below Base by $-0.87$. The same architectural change has opposite effects on opposite-quality data.
\item \textbf{Column-wise (data-quality drop):} both models lose BLEU on the noisier corpus, but Big loses twice as much ($-2.84$ vs $-1.41$). Capacity does not absorb noise; it amplifies the cost of noise.
\end{enumerate}

These are the same claim viewed two ways: capacity is a force multiplier whose sign depends on the supervision signal it is multiplying.

\subsection{Why v2 is noisy: direct evidence}
We claim v2's degradation is due to noise in the extra 20M pairs beyond the v1 head 10M. Three pieces of supporting evidence:

\paragraph{Spike-guard activations.} The loss-spike guard (\S\ref{sec:setup}) fires when a batch's loss exceeds $1.3\times$ the EMA --- intended to drop pathologically wrong batches. On Base v1.1's 122K-step run, it fires \textbf{0 times}. On Big v2's 416K-step run, it fires \textbf{12 times} ($\approx$~1 per 35K steps). 0 vs.\ nonzero is the direct signal: clean v1 contains no batches that loss-spike a competent model; v2 contains 12 such batches over its longer run. Section~\ref{sec:methodology} adds a methodology caveat about cross-capacity spike-guard comparisons.

\paragraph{Sample inspection.} Manual inspection of the 100 lowest-CometKiwi-22-scored pairs in v2 (where ``low'' is in the 0.05--0.3 range) shows the expected pathologies: untranslated source sentences echoed as ``translation'', misaligned sentence pairs, repeated boilerplate from web-scraped UN documents, partial translations, and translations of the wrong direction (fr~$\rightarrow$~en where en~$\rightarrow$~fr was expected). These pairs survive the per-pair length and ratio filters because the filters are token-counting heuristics that do not detect semantic misalignment.

\paragraph{Schedule was not the cause.} An earlier v2 attempt with 200K steps (linear $2\times$ over v1's 100K) underfit badly --- training loss had not reached its v1 floor by the schedule end. We rejected that attempt and used 600K (a $4\times$ schedule for $3.2\times$ data, with proportional warmup), which trains to a tight loss floor. The headline 600K-step Base v2 result is post-convergence, not under-training.

\subsection{Big v2 was halted early: was that the right call?}
Big v2 was planned for 800K steps and halted at 416K. The halt rule was: a single eval drops below the running best AND the run has been pathologically slow to find new bests. At halt, Big v2's averaged BLEU was 33.03 / 28.91, well below Base v2's 33.90 / 29.23 already established at the same wall-clock.

Could continued training to 800K have lifted Big v2 above Base v2? The training-curve trajectory (Figure~\ref{fig:big_v2_curves}) makes this implausible: the late-training BLEU oscillates within $\pm 0.5$ around 27.7--28.0 from step 300K through 416K with no upward trend, while training loss plateaus at $\sim$~3.0. Continued steps would have continued to reduce loss on noisy patterns and produced no further BLEU gain. We accept the halt as the conservative choice and note that Big v2's true asymptote could lie a few BLEU below the reported 33.03 (further noise fitting), not above.

\subsection{Combined prescription so far}
\S\ref{sec:enfr_v1}--\S\ref{sec:enfr_v2} together argue:

\begin{quote}
\textbf{Filter first, scale second.} The v1$\rightarrow$v2 corpus expansion sees both models lose, with the larger model losing more. The v1.1 capacity gap ($+0.56$) only emerges when the data is clean. Spending extra capacity on noisy data is strictly counterproductive at this scale.
\end{quote}

\S\ref{sec:ende} replicates the pattern at smaller scale (4.5M pairs, en-de). \S\ref{sec:sft} then asks whether v2's noise can be filtered after the fact via QE-filtered fine-tuning --- a natural follow-up that produces a different kind of failure.
